# JLPT N2 Trainer App 项目总结与修改建议

## 项目总结

这个项目当前不是“JLPT N2 题库 App”那么简单，而是一个本地优先的 N2 训练工作台。它的核心价值在于把“训练 -> 记录 -> 识别弱点 -> 推荐下一轮训练”做成了闭环，而不是只提供静态题目。入口和主状态都比较集中，应用从 [AppRoot.tsx](./src/app/AppRoot.tsx) 启动，通过 [ProgressProvider.tsx](./src/app/providers/ProgressProvider.tsx) 管理全局进度，再由 [AppNavigator.tsx](./src/app/navigation/AppNavigator.tsx) 用轻量自定义路由切换各训练页面。

从代码看，主要用户是已经开始系统备考 JLPT N2、愿意持续练习的人，而不是只想偶尔刷几题的路人。当前核心流程是：首页读取进度和 backlog，给出今天建议；用户进入 9 种训练模式之一；训练结果沉淀为 `sessionsByDay`、`wrongAnswers`、`weaknessSignals`、`studyWeaknesses`；首页再把这些信号转成新的推荐和薄弱点摘要。模式定义本身已经比较完整，集中在 [training.ts](./src/domain/models/training.ts) 和 [trainingModes.ts](./src/data/seed/trainingModes.ts)。

当前已经实现的核心能力包括：

- 9 个训练模式，覆盖文法、词汇、读解、听力、错题回收、官方词卡。
- 首页推荐逻辑、周目标、日目标、连续天数、能力分布。
- 文法/词汇错题回收机制。
- 读解/听力独立薄弱点信号机制。
- 学习包“不稳项”回流机制。
- AsyncStorage 本地持久化与历史裁剪。
- 官方词卡独立学习流程，不和普通学习包混用。

这版更像“可持续迭代的高保真产品原型”，已经明显超过 demo 和一次性 MVP，但还没有到可长期放量的产品阶段。原因是训练闭环已经成立，领域建模也比较完整，但内容生产、资源治理、测试和工程稳定性还没跟上。README 现在和代码主线基本一致，但有一处需要保守表述：README 说“听力案例 8 组”，这和 [listeningCases.ts](./src/data/seed/listeningCases.ts) 一致，不过其中有一部分是复用官方音频的占位训练案例，不等于 8 组全部是独立高质量官方改编内容。

## 业务层分析

这个项目当前最核心的业务价值，不是“题目多”，而是“让用户知道下一步该练什么”。首页推荐、薄弱点摘要、错题回收和学习包回流，构成了真正的用户留存点。用户之所以可能持续使用，不是因为有文法题或者词汇卡，而是因为系统在替他决定优先级，这一点体现在 [dashboardService.ts](./src/domain/services/dashboardService.ts) 和 [coachService.ts](./src/domain/services/coachService.ts)。

高价值功能主要有：

- 首页动态推荐，而不是模式列表。
- 错题回收和弱点聚合。
- 读解/听力不再被硬塞回文法错题队列，而是有独立信号。
- 学习包中的“不稳项”会重新影响后续训练。
- 官方词卡模式做了独立流程，避免和普通学习包混淆。

相对低价值或短期感知不强的部分有：

- 视觉层面的统一升级本身不是留存核心，只是辅助。
- “模式数量多”不一定天然提升价值，反而可能加重认知负担。
- 官方词卡模式如果来源和覆盖度不足，容易显得像附加功能而不是主力路径。

当前产品链路最大的几个问题：

1. 内容深度和系统深度不匹配。系统已经做成了“教练型训练器”，但内容规模还不足以支撑长期使用。4 篇读解、8 组听力、有限的 seed 题库，很快会见底。
2. 推荐逻辑成立，但缺少长期目标拆解。现在更多是在做“今天练什么”，还没有把“7 月 N2 通过”拆成阶段计划、能力地图和周节奏策略。
3. 训练结果的正反馈还偏弱。用户完成一轮后会被记录，但缺少更明确的“我为什么进步了”的反馈，比如按弱点类型的下降趋势、某类题稳定度上升。
4. 学习内容来源治理不够强。特别是官方词卡、扩展词库、听力占位案例混在一起时，用户很难判断哪些是核心主路径、哪些是补充内容。
5. 长期使用场景里缺少“重复打开理由”。现在更像一个强结构训练器，但还缺日更感、复习节奏提醒、阶段性里程碑和内容滚动供给。

如果目标是让用户长期学习 JLPT N2，这个产品还缺：

- 更明确的阶段学习计划。
- 更大量且可持续扩充的内容。
- 更强的复习节奏机制。
- 更清晰的“进步可视化”。
- 更少但更稳定的主训练路径定义。

按优先级排序，我会给业务迭代建议：

1. 先把“主路径”收紧成 3 条：今日推荐、错题/弱点回收、阶段复习。现在模式很多，但用户真正需要的是少量高频决策。
2. 补“长期计划层”。把首页从“今日建议”升级为“今日建议 + 本周任务 + 当前阶段目标”。
3. 优先扩内容，而不是继续加模式。现阶段最大的业务短板是内容供给。
4. 增加“进步证据”展示。比如本周薄弱点变化、错题压降、读解/听力稳定项回升。
5. 为官方词卡和扩展词库加清晰标签。区分官方、模拟、扩展、自建，避免用户对内容可信度产生误解。

## 代码层分析

整体结构是清晰的，而且比很多小型 React Native 项目更有边界感。`app / features / domain / data` 这种分层是成立的，尤其领域模型和服务层并没有直接埋在页面里，这是优点。像 [progress.ts](./src/domain/models/progress.ts)、[trainingContent.ts](./src/domain/models/trainingContent.ts)、[progressService.ts](./src/domain/services/progressService.ts) 这组文件已经把核心业务抽出来了，这点应该保留。

设计得不错的地方：

- 训练模式和模式分流集中定义，新增模式时心智路径明确。
- `ProgressProvider` 只做状态装配，不把完整业务规则塞进 Provider。
- `dashboardService` 和 `coachService` 把推荐逻辑从 UI 中剥离出来。
- `wrongAnswerClassifier.ts` 把错误标签映射为弱点类型，这对后续扩展很重要。
- `APP_CONFIG` 在 [constants.ts](./src/config/constants.ts) 中集中定义，避免魔法数字散落。

会在后续迭代里成为负担的地方：

1. `progressService.ts` 体积已经明显过大，承担了规范化、迁移、排序、打分、记录、裁剪、查询等过多职责。它现在是系统核心，但也快变成“万能服务”。
2. seed 数据文件过大且直接嵌在 TS 中，尤其 [readingPassages.ts](./src/data/seed/readingPassages.ts)、[listeningCases.ts](./src/data/seed/listeningCases.ts)、[officialVocabDecks.ts](./src/data/seed/officialVocabDecks.ts)。这会拖累维护、审阅、编码稳定性和内容治理。
3. 自定义导航在当前规模还能接受，但随着模式继续增加、回流路径继续变复杂，[AppNavigator.tsx](./src/app/navigation/AppNavigator.tsx) 会越来越重。
4. 页面和领域的边界大体清楚，但“会话流程逻辑”可能仍有重复分散在多个 screen 里。这个我基于文件结构推断存在，建议后续重点检查结果页、退出逻辑、记录逻辑是否重复。
5. 当前没有看到测试结构。对这种“规则驱动型推荐系统”，最该有的其实不是 UI 测试，而是服务层单测，尤其是推荐逻辑和进度迁移逻辑。

潜在重构重点模块：

- [progressService.ts](./src/domain/services/progressService.ts)
  - 应拆成 `progressNormalization`、`progressQueries`、`progressRecording`、`progressScoring` 一类的小模块。
- [AppNavigator.tsx](./src/app/navigation/AppNavigator.tsx)
  - 现在靠一组 `if` 分发 screen，模式再增多时可维护性会继续下降。
- seed 目录下的大文件
  - 应逐步转向 JSON / 分包内容 / 生成脚本，而不是继续在 TS 文件里硬写大段内容。
- [officialVocabDecks.ts](./src/data/seed/officialVocabDecks.ts)
  - 当前内容管理压力已经开始显现，而且文本编码问题风险较高。
- [studyPacks.ts](./src/data/seed/studyPacks.ts)
  - 这里混合了静态内容、映射逻辑、阶段切片逻辑，后续可以再拆。

比较明显的风险：

- 文本资源维护风险。你仓库里已经能看到部分 seed 文件和输出里存在乱码迹象，这不是业务逻辑问题，但会直接影响内容可信度和维护效率。
- 内容与代码耦合过深。现在“改内容”几乎等于“改代码”，这会放大每次内容迭代成本。
- 缺少规则测试。推荐优先级、薄弱点聚合、复习回流这类规则一旦改坏，很难靠肉眼稳定验证。
- 持久化模型还在演进中。你已经做了迁移兼容逻辑，这是好事，但也说明状态模型未来还会继续变，越往后越需要专门的迁移测试。

不过也要明确一点：当前项目规模还没大到需要过度架构设计。现在最该做的不是上 Redux、建复杂 DDD 框架或拆十几层 abstraction，而是把已有的领域模型继续守住，把超大的服务文件和内容文件拆开，再补测试。

## 建议清单

### 业务修改建议

- 建议项：把首页主路径收紧为“今日推荐 / 薄弱点回收 / 阶段复习”
- 解决的问题：模式多但用户决策成本高
- 为什么优先：这是留存入口，直接影响每天是否能快速开始
- 修改成本：中
- 预期收益：提升启动效率，减少用户犹豫
- 是否建议现在做：建议现在做

- 建议项：增加“阶段计划”层，而不只做今日推荐
- 解决的问题：产品缺少长期目标感
- 为什么优先：N2 备考是长周期任务，只有日任务不足以支撑长期留存
- 修改成本：中
- 预期收益：让产品从训练器更接近真正教练
- 是否建议现在做：建议现在做

- 建议项：优先扩充高质量内容，而不是继续加新模式
- 解决的问题：系统深度已经超过内容供给深度
- 为什么优先：内容不足会比功能不足更快导致流失
- 修改成本：高
- 预期收益：显著提升可持续使用时长
- 是否建议现在做：建议现在做

- 建议项：增加“进步证据”展示
- 解决的问题：用户做完训练后缺少成就反馈
- 为什么优先：长期学习产品需要可见进步来维持复访
- 修改成本：中
- 预期收益：增强复购式打开动机
- 是否建议现在做：建议现在做

- 建议项：为内容来源建立清晰标签体系
- 解决的问题：官方、模拟、扩展内容边界不够清楚
- 为什么优先：会直接影响用户对内容可信度的判断
- 修改成本：低
- 预期收益：提升信任感，减少认知混乱
- 是否建议现在做：建议现在做

### 代码修改建议

- 建议项：拆分 [progressService.ts](./src/domain/services/progressService.ts)
- 解决的问题：单文件职责过多，未来修改风险大
- 为什么优先：这是全局核心服务，继续膨胀会拖慢所有迭代
- 修改成本：中
- 预期收益：提升可维护性和测试可写性
- 是否建议现在做：建议现在做

- 建议项：给推荐和进度规则补服务层测试
- 解决的问题：规则改动容易引入静默回归
- 为什么优先：这是项目最有价值的核心逻辑
- 修改成本：中
- 预期收益：显著降低后续改动风险
- 是否建议现在做：强烈建议现在做

- 建议项：把大体量 seed 内容逐步迁到数据文件或生成流程
- 解决的问题：内容维护和代码维护耦合过深
- 为什么优先：内容会持续增长，越晚拆越重
- 修改成本：中
- 预期收益：提升内容迭代效率，降低 merge 冲突和编码问题
- 是否建议现在做：建议现在做

- 建议项：重构 [AppNavigator.tsx](./src/app/navigation/AppNavigator.tsx) 的 screen 分发表达
- 解决的问题：模式增多时路由分发可读性下降
- 为什么优先：虽然现在还能撑住，但已经有继续膨胀趋势
- 修改成本：低
- 预期收益：提升可读性，降低新增模式的改动成本
- 是否建议现在做：可以现在做

- 建议项：把学习内容管理逻辑从 [studyPacks.ts](./src/data/seed/studyPacks.ts) 和 [officialVocabDecks.ts](./src/data/seed/officialVocabDecks.ts) 中继续拆开
- 解决的问题：内容、映射、分页/切片逻辑混在一起
- 为什么优先：后续扩内容时会越来越难维护
- 修改成本：中
- 预期收益：内容结构更稳，修改更安全
- 是否建议现在做：建议现在做

## 下一轮最值得做的 5 件事

1. 给推荐、弱点聚合、持久化迁移补单元测试。
2. 拆分 `progressService`，把查询、记录、归一化、打分拆开。
3. 首页增加“阶段计划 / 本周目标”层，强化长期学习感。
4. 扩充高质量读解与听力内容，优先补主路径内容深度。
5. 整理 seed 内容和来源标签，降低内容维护与可信度风险。
