## Problem Statement

JLPT N2 备考 app 内容量不足（刷来刷去就那些题），且缺乏智能辅助——错了题不知道为什么错，没有方向感和进度感。目标是加入 AI 功能，让学习更有针对性。

## What Makes This Cool

错了题，点一下"为什么我错了"——AI 直接告诉你：你的错误模式是什么、这个干扰项为什么骗到你了、下次要注意什么。不是通用解释，是针对你这次错误的精准分析。同时，数据结构为未来的 AI 每日学习计划铺好路。

长期愿景（Codex 提出的 "Shadow Examiner"）：app 学习你的错误签名，专门生成针对你弱点的对抗性题目，每周输出"如果今天考试，这些会让你挂科"报告。

## Constraints

- 个人项目 + 学习编程，优先能跑起来、能学到东西
- 需要支持切换 AI 提供商（国内用 DeepSeek，国外用 Claude）
- 不引入后端（现阶段 API Key 写 config，后续产品化时再加）
- 现有题库量少，AI 功能不能依赖海量内容

## Premises

1. App 主要目标是通过 JLPT N2 考试，不是泛用日语学习工具
2. 第一步做 **AI 错题解释**（最快落地），**学习计划数据结构**同步设计但用本地逻辑填充，AI 出题放后期
3. 第一版不必完美，能用就行，迭代推进

## Cross-Model Perspective

Codex（gpt-5.4）独立分析后与 Claude 完全一致：先做 AI 错题解释，数据结构并行设计。

Codex 补充的两个关键点：
- **工具推荐**：`ts-fsrs`（FSRS 算法的 TypeScript 实现），可替换现有手写优先级权重逻辑，获得更科学的间隔复习调度
- **"Shadow Examiner" 愿景**：V2 方向不是"出更多题"，而是"专门出你容易错的题型"——分析错误签名，生成对抗性 mini-mock，每周出"今日考试会挂科的点"报告

## Approaches Considered

### Approach A: 最小可行 AI（周末能跑起来）
- Summary: 在错题回顾界面加"为什么我错了"按钮，直接调 Claude/DeepSeek API 返回解释
- Effort: S（1-2天）
- Risk: Low
- Pros: 立竿见影；学 API 调用的最佳起点；不改现有架构
- Cons: API Key 暴露客户端（个人用无妨）；无缓存每次等待；不留后路
- Reuses: WrongReviewScreen.tsx, WrongAnswerItem 现有结构

### Approach B: AI 解释 + 学习计划数据结构（推荐）
- Summary: A 方案基础上，同步在 ProgressState 定义 GeneratedDailyPlan 类型，用本地逻辑填充，为后续 AI 计划预留接口
- Effort: M（3-5天）
- Risk: Low
- Pros: 马上能用 AI 解释；数据结构设计好后接 AI 计划只换实现；可引入 ts-fsrs
- Cons: 比 A 多几天，但每一步都值得做
- Reuses: progressService.ts, coachService.ts, progress.ts 现有类型

### Approach C: Shadow Examiner（长期愿景）
- Summary: 分析错误模式，生成针对性对抗题目，每周出考试风险报告
- Effort: XL（数周）
- Risk: High
- Pros: 真正差异化；如果创业这是护城河
- Cons: 技术复杂；内容质量难控制；题库太小时效果差

## Recommended Approach

**方案 B**。

A 太短，不留后路；C 太早，地基未稳。B 是刚好——AI 错题解释当天可用，数据结构为未来铺路。

### 具体实现步骤

**Step 1：创建 aiCoachClient.ts（适配层）**

```
src/services/aiCoachClient.ts
```

接口设计（提供商无关）：
```ts
type WrongAnswerExplanationParams = {
  question: string;
  choices: string[];
  selectedChoice: number;
  correctChoice: number;
  tags: string[];
  modeId: string;
  wrongCount: number;
  explanation: string; // 现有静态解释
};

type WrongAnswerExplanation = {
  mistakePattern: string;    // 你的错误模式
  whyDistractorFooled: string; // 为什么这个干扰项骗到你了
  watchNextTime: string;     // 下次要注意什么
};

getWrongAnswerExplanation(params): Promise<WrongAnswerExplanation>
```

内部实现：读 config 里的 `AI_PROVIDER`（`'claude' | 'deepseek'`），切换对应 base URL 和 auth header。DeepSeek API 与 OpenAI 格式兼容，切换几乎零成本。

**System Prompt 模板（直接用于 API 调用）：**

```
You are a JLPT N2 Japanese exam coach. A student answered a question incorrectly.

Question: {question}
Choices: {choices joined by " / "}
Student chose: {choices[selectedChoice]}
Correct answer: {choices[correctChoice]}
Tags: {tags joined by ", "}
Times this student has gotten this wrong: {wrongCount}

Always respond in Chinese (Simplified).
Return ONLY valid JSON with exactly these three fields:
{
  "mistakePattern": "one sentence: what error pattern this reveals about the student",
  "whyDistractorFooled": "one sentence: why the wrong choice looked plausible",
  "watchNextTime": "one sentence: specific thing to remember to avoid this next time"
}
No preamble. No markdown. Only the JSON object.
```

**Step 2：在 `src/domain/models/progress.ts` 加类型**（与 ProgressState 同文件，保持架构一致）

```ts
export type AiWrongAnswerExplanation = {
  mistakePattern: string;
  whyDistractorFooled: string;
  watchNextTime: string;
  generatedAt: string; // ISO 8601
};
```

**前置确认：缓存 key 用 `questionId`**

`WrongAnswerItem` 已有 `questionId: string` 字段（在 `src/domain/models/trainingContent.ts` 可确认）。用 `questionId` 作为缓存 key，无需新增字段。

**Step 3：在 ProgressState 加缓存**

```ts
// progress.ts — ProgressState 类型里加：
aiExplanationCache: Record<string, AiWrongAnswerExplanation>; // key: WrongAnswerItem.questionId
```

缓存清理：`progressService.ts` 的 `completeWrongReviewSession` 函数在移除已掌握的 `WrongAnswerItem` 时，同步删除对应的 `aiExplanationCache[id]` 条目，避免 AsyncStorage 无限膨胀。

避免重复调用 API，节省费用。

**Step 4：在 WrongReviewScreen.tsx 加 UI**

保留现有静态 explanation，在下方加一个 CTA：

> 「为什么我错了？」（按钮）

点击后展示三行：
- 错误模式：...
- 干扰项为何有效：...
- 下次注意：...

加载状态用 ActivityIndicator。超时设定 **10 秒**（`AbortController` + `setTimeout`）。错误处理分三层：
- **网络/超时错误**：显示「获取解释失败，请稍后重试」
- **JSON 解析失败**（LLM 返回非标准 JSON）：同上，并在开发模式下 `console.warn` 原始响应
- 所有错误均静默降级，不弹弹窗，不影响答题主流程

**Step 5：并行定义 GeneratedDailyPlan（不调 AI，只定数据结构）**

```ts
// progress.ts
export type DailyStudyItem = {
  modeId: TrainingModeId;
  reason: string;          // 为什么推荐这个
  priority: 'urgent' | 'normal';
  estimatedMinutes: number;
};

export type GeneratedDailyPlan = {
  date: string;
  items: DailyStudyItem[];
  generatedBy: 'local' | 'ai'; // 先填 'local'，后期换 'ai'
  examCountdown?: number;       // 距考试天数（可选）
};
```

`coachService.ts` 的 `getDashboardWeaknessSnapshot()` 函数（当前返回 `DashboardWeaknessSnapshot`，包含 `planSteps: CoachPlanStep[]`）——在其中新增一个并行函数 `getGeneratedDailyPlan()` 返回 `GeneratedDailyPlan`，基于现有本地逻辑填充。不修改现有 `getDashboardWeaknessSnapshot()`，避免影响 Dashboard 现有显示。

**注意：** `TodayPlanCard` 的 `todayPlan` prop 是 `TrainingMode[]`（推荐模式列表），与 `GeneratedDailyPlan`（弱点学习计划）是两套独立系统，不要混淆，不需要修改 `TodayPlanCard`。

**注意：** `GeneratedDailyPlan` 类型定义和 `buildCoachPlan` 迁移是**独立故事**，可在 AI 解释功能跑通后再做，不阻塞 Step 1-4。

### AI 提供商切换

// src/config/constants.ts 加到 APP_CONFIG 对象里

国内用 DeepSeek：改 `AI_PROVIDER: 'deepseek'`，换 API Key，完成。

**⚠️ 安全注意**：`EXPO_PUBLIC_` 前缀意味着 API Key 会被打包进 JS bundle。Expo Go 本地开发完全没问题。**如果将来用 EAS Build 打包分发给别人**，必须改成通过自建后端代理调用，不能把 Key 留在客户端。这是个人用 → 产品化时的必做改造。

## Open Questions

1. 是否需要考试倒计时功能？（影响 `GeneratedDailyPlan.examCountdown` 字段的优先级，可后期加）
2. ts-fsrs：**不在本次迭代范围内**。等 AI 计划功能上线、题库量达到 200+ 时再引入，届时需要为每道 `WrongAnswerItem` 新增 FSRS 卡片状态字段。

## Success Criteria

- 在 WrongReviewScreen 错误题目下，点击按钮 **10 秒内**（WiFi 环境）返回 AI 解释
- 解释内容包含：错误模式 / 干扰项分析 / 下次注意点 三个字段
- 切换 AI_PROVIDER 配置后，不需要改其他任何文件
- `GeneratedDailyPlan` 和 `DailyStudyItem` 类型定义完成，`coachService.ts` 新增 `getGeneratedDailyPlan()` 函数返回该类型（现有 `getDashboardWeaknessSnapshot()` 保持不动）
