## 一、项目总结

### 1.1 项目定位

面向 JLPT N2 备考的本地训练工具。核心是"训练 → 记录错误 → 识别弱点 → 推荐下一步"的闭环。无后端，数据全部存在 AsyncStorage（key: `jlpt-n2-trainer-state-v1`）。

### 1.2 核心使用流程

```
首页（DashboardScreen）
  → 查看推荐模式 + Coach 弱点摘要
  → 进入某模式（ModeDetailScreen → 导航分发）
  → 完成训练（DrillSession / StudyPack / Reading / Listening / WrongReview）
  → 写入 ProgressState（wrongAnswers / weaknessSignals / studyWeaknesses）
  → 回首页，首页重新计算推荐和弱点聚合
```

### 1.3 已实现的核心能力

- **三层弱点追踪**：`wrongAnswers`（刷题错题）+ `weaknessSignals`（读解/听力题级信号）+ `studyWeaknesses`（学习包自查）
- **跨模式弱点聚合**：`coachService.aggregateWeaknesses` 把三个来源合并排序，输出 Coach 摘要和攻克计划
- **优先级排序的错题回收队列**：`getPrioritizedWrongAnswersForMode` 有实际的加权优先级算法，非简单 FIFO
- **动态学习包**：当前 session 会把 active 弱项置顶合并到 basePack 前面
- **首页 Insight 状态机**：`getDashboardInsight` 有 5 种状态，根据当天进度、积压、周目标给出差异化引导

### 1.4 当前阶段判断

**介于"功能完整的 MVP"和"可持续迭代版本"之间，更偏向前者。**

- 核心训练闭环已通，数据模型清晰
- 但内容量（seed data）是手工维护的静态 TS 文件，扩展成本高
- 没有任何自动化测试，迭代风险随复杂度上升
- 官方词卡模式的重要分类（文字・語彙、読解）在代码里标记为 `status: 'pending'`，功能入口开着但内容是空的

### 1.5 README vs 代码不一致之处

1. README 说"9 个训练模式都能稳定进入"，但 `OfficialVocabMemoryScreen` 里 `pending` 状态的词卡包进去只能看到空状态，不算真正可用。
2. 其余内容与代码基本一致。

## 二、业务层分析

### 2.1 核心价值

**真正的价值点只有一个：跨模式弱点聚合 + 有依据的"下一步该做什么"。**

这是这个产品和普通题库 app 的本质区别。用户做完读解答错，首页能认出这是"读解证据定位"类弱点，并告诉用户下一步该做什么。这套逻辑在代码里是真实实现的，不是样子货。

### 2.2 用户留存判断

**短期会留，长期不确定，核心风险是内容量。**

读解、听力都是手工编写的静态素材。用户每天练，一两周内就会把全部读解/听力刷完，之后这两个模式就没有新内容可用，弱点系统也因为没有新错误输入而停止运转。

### 2.3 高价值 vs 低价值功能

**高价值（真正能帮用户）：**
- 首页 Coach 卡（弱点聚合 + 攻克计划）
- 错题回收（优先级排序真实有效，回收页解析质量高）
- 读解实战（内容质量好，有 evidence 字段和 choiceInsights）

**看起来多但价值存疑：**
- **官方词卡记忆模式**：文字・語彙 和 読解 词卡包是空的，入口是开的，新用户进去只看到"待导入"，造成负面印象。
- **学习包"不稳项"机制**：设计思路对，但没有时间间隔控制。同一天多次刷学习包，相同"不稳项"反复出现，没有遗忘-重现效果，用户会用印象代替真实回忆打分。
- **HeroCard Battle State**：只看 backlog 数量判断状态，不看正确率。用户 backlog 为 0 但正确率很低，也会显示 `goal_reached`，与实际情况不符。

### 2.4 产品链路主要问题（共 5 个）

**问题-01：内容会被耗尽**
读解、听力、刷题都是静态 seed 数据。没有内容扩充机制，用户刷完一遍之后系统没有新内容驱动，弱点系统也随之停转。这是当前最严重的留存风险。

**问题-02：错题回收的"掌握"是自我声明，没有验证**
`WrongReviewScreen` 里用户看完题目和解析，直接点"这题我已掌握"就移出队列。没有要求重新作答。系统无法区分"真的记住了"和"看一眼觉得自己懂了"。

**问题-03：学习包没有时间间隔，弱项被反复机械展示**
`studyWeaknesses` 只要 `active: true` 就每次置顶。用户一天内做多轮，每轮都看同一批"不稳项"，没有遗忘曲线触发，自查准确度会越来越低。

**问题-04：没有进步可见性**
首页能看到今天做了几轮、本周节奏、当前 backlog，但用户看不到"我的文法错题比上周少了多少"、"听力信息追踪类题有没有在改善"。弱点系统是单向的：只告诉你"还有这个弱点"，不告诉你"这个弱点在消退"。用户感知不到自己在变好，留存动力不足。

**问题-05：官方词卡 pending 分类挂在首页导航里**
文字・語彙和読解词卡包标记为 `pending`，但在 UI 上是可见可点击的入口。对新用户是糟糕的第一印象，增加了导航复杂度但没有交付价值。

### 2.5 业务迭代优先级

| 编号 | 问题 | 优先级 |
|---|---|---|
| B1 | 隐藏或明确标记 pending 词卡分类 | P0（立刻修） |
| B2 | 错题回收加重新作答验证 | P1 |
| B3 | 学习包弱项加最短复现间隔 | P1 |
| B4 | 弱点消退趋势展示 | P2 |
| B5 | 内容扩充机制（JSON 化 + 持续维护） | P2（不排入计划会一直欠） |

## 三、代码层分析

### 3.1 整体结构评价

**结构清晰，分层合理。** 在同等规模的项目中属于上游水准。

- `domain/models/`：纯类型，无逻辑
- `domain/services/`：纯函数，无副作用
- `features/`：只负责 UI 和调用 store
- `data/`：负责 seed 和持久化
- `app/`：负责启动、导航、Provider

### 3.2 做得好的地方（应该保留）

- **`ProgressProvider` 的单一 store + 纯函数 reducer 模式**：所有状态变更走 `progressService` 纯函数，Provider 只触发和持久化，逻辑可追溯。
- **`normalizeProgressState` 的 defensive loading**：加载时做字段校验和默认值填充，旧存档不会崩，是很多项目缺失的。
- **`MODE_CAPABILITY` 显式映射**（`dashboardService.ts`）：`Record<TrainingModeId, keyof CapabilityDistribution>`，类型安全，不依赖字符串前缀。
- **`APP_CONFIG` 集中配置 + `dateUtils` 工具提取**：魔法数字和重复日期工具已经处理，这两个重构到位了。
- **手动 Route 导航**：不引入 React Navigation 对当前规模是正确决策，9 个屏幕用 union type + 条件渲染已经够用。
- **`coachService` 的 `buildPlanSteps` 已有空数组 guard**：`if (focusItems.length === 0) return buildNeutralPlan()` 已到位。

### 3.3 会成为后续迭代负担的地方

**问题-C1：Feature Screen 文件体积大，StyleSheet 和逻辑内联**
`WrongReviewScreen.tsx` 640 行（StyleSheet 占后 300 行），`DrillSessionScreen.tsx` 更长。每次改功能要在长文件里翻找，样式和逻辑耦合在一起。

建议：StyleSheet 提取到同目录 `xxxStyles.ts`，核心状态逻辑提取成 `useXxxSession` hook。

**问题-C2：`aggregateWeaknesses` 三段循环逻辑高度重复**（`coachService.ts` 第 52-168 行）
三段 for 循环（wrongAnswers、weaknessSignals、studyWeaknesses）结构几乎一样，差别只是来源对象的字段名和 score 计算方式。如果以后新增弱点来源，就需要复制第四段。

建议：提取成通用的 `accumulateToMap(items, getScore, getErrorTypes, summaryMap)` 函数。

**问题-C3：Seed 数据是 TypeScript 文件，内容扩充依赖开发者介入**
`readingPassages.ts`、`listeningCases.ts`、`drillQuestions.ts` 都是手写 TS 常量数组。加一篇读解文章需要改代码、跑 Metro。非开发人员无法维护内容，这是内容扩充瓶颈的技术根源。

**问题-C4：学习包选取依赖 `initialSessionCount` 语义隐晦**（`StudyPackScreen.tsx` 第 49 行）
当天第 N 次完成该模式时，拿第 N 个 pack。`clearToday` 会导致学习包倒回第一个，用户感知不到。逻辑隐晦，与弱点系统叠加后行为难以预测。

**问题-C5：零测试覆盖**
`progressService` 的优先级算法、`coachService` 的弱点聚合、`dashboardService` 的 Streak 计算，全是核心业务逻辑，完全没有测试覆盖。任何改动都是裸奔。一旦 `normalizeProgressState` 需要迁移字段，无法验证是否引入了回归。

**问题-C6：`DAILY_TARGET_SESSIONS` 语义不一致**
`getTodayPlan` 用它限制推荐数量，`getDashboardInsight` 用它作为"今日达标线"，两个语义不同但用同一个常量。如果以后想让推荐显示 5 个但达标线是 3 个，需要拆开。

### 3.4 可维护性 / 扩展性汇总

| 维度 | 评价 | 说明 |
|---|---|---|
| 可维护性 | 中 | 分层合理，但 Feature Screen 体积大 |
| 内容扩展性 | 低 | Seed 数据结构不支持外部维护 |
| 功能扩展性 | 中 | 新增模式需改 `training.ts`、`AppNavigator`、`progressService`，路径清晰但步骤多 |
| 领域层耦合度 | 低 | 服务层纯函数，解耦好 |
| UI 层耦合度 | 中 | Screen 文件内 StyleSheet 和逻辑混在一起 |
| 测试风险 | 高 | 0 覆盖率，改核心逻辑无保障 |

## 四、建议清单

### 4.1 业务修改建议

**[B1] 隐藏或明确标记未完成的官方词卡分类**
- 解决的问题：新用户进入官方词卡模式，看到"待导入"标签，体验断裂
- 为什么优先：功能开放但内容不在线，等于在首页展示了一个不能用的入口
- 修改成本：低（UI 判断 `status === 'pending'` 时隐藏或降级展示即可）
- 预期收益：消除新用户困惑，降低第一印象受损风险
- **建议现在做：是**

**[B2] 错题回收加"重新作答"步骤**
- 解决的问题：用户看完解析直接点"掌握"，无实际验证，弱点清零是虚假的
- 为什么优先：错题回收是整个弱点闭环的"验收环节"，验收不严则闭环无意义
- 修改成本：中（WrongReviewScreen 需增加答题态，并基于是否答对更新决策逻辑）
- 预期收益：弱点数据更准确，二次回收率下降，学习效果可感知
- **建议现在做：是**

**[B3] 学习包弱项加最短复现间隔（≥ 4 小时）**
- 解决的问题：同一天内多次刷学习包，同批"不稳项"反复出现，用户用印象代替回忆打分
- 为什么优先：不解决，`studyWeaknesses` 数据质量会越来越差
- 修改成本：低（在 `getActiveStudyWeaknesses` 里加时间过滤，距 `lastUnstableAt` < 4 小时则跳过）
- 预期收益：学习包自查结果更真实，弱项清除速度更有参考意义
- **建议现在做：是**

**[B4] 弱点消退趋势展示**
- 解决的问题：用户只能看到 backlog 数量，看不到"我在变好"
- 为什么优先：这是长期留存的核心驱动，用户需要感知到"某类错误在减少"
- 修改成本：中（需在 ProgressState 里加历史计数，或在 Dashboard 展示 weaknessSignals 的 active/resolved 比例）
- 预期收益：用户留存提升，"训练有效"的感知强化
- **建议现在做：否（P1，排入下轮）**

**[B5] 内容扩充机制（Seed 数据 JSON 化 + 独立维护流程）**
- 解决的问题：读解/听力题目耗尽后没有新内容，用户两周内流失
- 为什么优先：这是长期存活的底层条件
- 修改成本：高（Seed TS 文件改造成 JSON，更新 import 方式，类型校验改为 JSON schema）
- 预期收益：可以持续扩充内容而不依赖开发者改代码
- **建议现在做：否（需专项规划，但不要拖超过一个月）**

### 4.2 代码修改建议

**[C1] 为核心服务层加单元测试**
- 解决的问题：`progressService`、`coachService`、`dashboardService` 的核心逻辑 0 覆盖，任何改动都有回归风险
- 为什么优先：现在逻辑已经稳定，是加测试的最佳时机；等功能继续扩展，测试越来越难补
- 修改成本：中（引入 Jest/Vitest，为三个 service 各写 10-15 个测试用例）
- 预期收益：改核心逻辑时有基本安全网，特别是 `normalizeProgressState` 迁移场景
- **建议现在做：是**

**[C2] Feature Screen 的 StyleSheet 提取到独立文件**
- 解决的问题：`WrongReviewScreen.tsx`、`DrillSessionScreen.tsx` 等文件超 600 行，逻辑和样式混杂
- 为什么优先：目前还不是很痛，但随着功能迭代会越来越差
- 修改成本：低（直接移文件，无逻辑变更）
- 预期收益：每个 screen 文件聚焦在业务逻辑上，可读性提升
- **建议现在做：否（可顺手做，不用专门排期）**

**[C3] 重构 `aggregateWeaknesses` 的三段重复循环**
- 解决的问题：`coachService.ts` 第 52-168 行三段结构几乎相同，新增弱点来源需要复制整段逻辑
- 为什么优先：如果后续新增弱点类型，会直接踩到这里
- 修改成本：低-中（提取通用聚合函数，三段替换为三次调用）
- 预期收益：新增弱点来源只需一次调用，不需要复制逻辑
- **建议现在做：否（下次动 coachService 时顺手做）**

**[C4] Seed 数据迁移到 JSON 文件**
- 解决的问题：内容扩充依赖开发者改 TS 文件，非开发人员无法参与
- 为什么优先：和业务建议 B5 对应，是内容可持续的技术前提
- 修改成本：高（需迁移所有 seed 数据，更新 import，类型校验改为 JSON schema）
- 预期收益：内容维护独立于代码，可以单独维护题库文件
- **建议现在做：否（和 B5 一起规划）**

**[C5] 拆分 `DAILY_TARGET_SESSIONS` 的推荐数量和达标线两个语义**
- 解决的问题：同一常量承载两个可能分化的语义，未来调整时需要拆开
- 为什么优先：目前还没问题，但留个隐患
- 修改成本：低（加一个新常量，改引用）
- 预期收益：配置语义清晰，未来调整推荐数量不影响达标判断
- **建议现在做：否（有空就顺手拆）**

## 五、下一轮最值得做的 5 件事（按优先级）

**第 1 件：隐藏 pending 状态的官方词卡分类 [B1]**
修改成本极低，但影响第一印象。不修就一直在首页挂着一个残缺的功能入口。

**第 2 件：错题回收加重新作答验证 [B2]**
核心闭环的最后一公里。当前"掌握"是用户自报，弱点数据没有实际验证。加了这步，弱点清零才有意义。

**第 3 件：为核心 service 层加单元测试 [C1]**
逻辑已稳定，现在不加以后越来越难补。特别是弱点聚合算法和 Streak 计算，出了 bug 用户完全感知不到（数据静默错误）。

**第 4 件：学习包弱项加时间间隔过滤 [B3]**
一行过滤逻辑可以解决，但不解决会持续污染 studyWeaknesses 数据质量。

**第 5 件：启动内容 JSON 化 + 题库扩充规划 [B5 + C4]**
这是留存的底线。读解和听力题目数量有限，两周高频用户就能刷完。这件事修改成本高，必须主动排入计划。
