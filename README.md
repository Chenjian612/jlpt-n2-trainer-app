# JLPT N2 Trainer App

## 项目定位

这是一个面向 JLPT N2 备考者的轻量训练 App，目标不是直接堆完整题库，而是先把每天的训练节奏、模式入口和打卡反馈搭起来，降低“今天该练什么”的决策成本。

当前版本更接近一个“每日训练指挥台”：

- 用固定的 8 种训练模式组织学习内容
- 用首页 dashboard 展示今日计划和本周节奏
- 用本地持久化记录每天完成了哪些训练
- 用模式详情页告诉用户这一轮训练该怎么做

一句话概括：

> 把 JLPT N2 的长期冲刺拆成每天可执行的 3 步训练，并用本地打卡与节奏反馈帮助持续推进。

## 当前产品形态

当前项目已经具备一个可演示的 MVP 骨架，但还没有接入真正的题目流和错题流。

### 用户打开 App 后会看到什么

首页 dashboard 目前包含：

- 今日完成进度
- 当前连练和最佳连练
- 本周训练进度
- 今日推荐的 3 项训练
- 8 大训练模式入口

点击某个训练模式后，会进入模式详情页，展示：

- 这个模式的目标和定位
- 今日训练步骤
- 检查清单
- 完成标准
- 今日打卡按钮

### 当前 8 个训练模式

- 文法闯关
- 文法记忆包
- 词汇刷题
- 词汇记忆包
- 读解实战
- 听力陷阱分析
- 文法错题回收
- 词汇错题回收

这些模式已经有清晰的文案、时长、重点、步骤和复盘建议，但目前仍是静态内容配置，尚未接入真实做题或学习数据。

## 当前已实现的内容

### 1. 页面骨架

项目当前只有两层主界面：

- dashboard 首页
- mode detail 训练模式详情页

导航使用的是轻量的本地状态切换，而不是完整的路由库。这对于当前原型阶段是合理的，结构简单，维护成本低。

### 2. 本地进度系统

项目已经具备基础进度管理能力，数据保存在本地 AsyncStorage 中，当前记录内容包括：

- 每周目标
- 每天完成了哪些训练模式
- 今日完成数
- 本周完成数
- 累计训练场次
- 当前连练天数
- 最佳连练天数

历史数据目前只保留最近 45 天。

### 3. 产品文案和模式结构

8 个训练模式的元数据已经定义得比较完整，每个模式都有：

- 标题和短标题
- 模式副标题
- 简要描述
- 推荐时长
- 训练重点
- 来源标签
- 详情页介绍文案
- 训练步骤
- 检查清单
- 完成标准
- 复盘建议
- 视觉配色

这意味着产品层面的“训练框架”已经成型，后续重点不是再写更多说明文案，而是把真实训练内容和交互接进来。

### 4. 初步分层架构

仓库已经做了较清晰的前端分层：

- `src/app`：应用根、导航、Provider
- `src/features`：页面、组件、view model
- `src/domain`：领域模型与纯逻辑服务
- `src/data`：本地存储、仓储、seed 数据
- `src/components`：公共组件
- `src/theme`：视觉 token

这套结构对后续扩展题库、错题、推荐逻辑、同步能力都比较友好。

## 当前还没有实现的内容

这是当前项目最重要的现实判断：它已经是一个完整的训练骨架，但还不是一个完整的学习产品。

目前缺少的核心能力包括：

- 真实题目数据
- 真正的做题交互
- 学习包内容浏览与完成逻辑
- 错题来源和错题队列
- 自动完成记录
- 个性化推荐逻辑
- 用户账号与云同步
- 内容更新机制
- 自动化测试
- 完整的工程说明文档

也就是说，当前版本主要解决的是“训练编排和节奏反馈”，还没有解决“内容交付、做题反馈和复习闭环”。

## 技术栈

## 当前代码结构导向

如果从代码职责理解这个项目，可以这样看：

### App 层

`App.tsx` 只负责挂载 `AppRoot`。

`src/app/AppRoot.tsx` 负责包裹全局 `ProgressProvider`，并挂载状态栏和导航。

`src/app/navigation/AppNavigator.tsx` 负责：

- 等待进度数据 hydration
- 在 dashboard 与 mode detail 之间切换

### 数据与状态层

`src/app/providers/ProgressProvider.tsx` 负责：

- 初始化进度状态
- 从本地存储加载数据
- 在状态变化后自动持久化
- 对外提供 toggle、clear、setWeeklyGoal 等能力

`src/data/repositories/progressRepository.ts` 负责：

- 读写本地进度数据

`src/data/storage/asyncStorageClient.ts` 负责：

- 对 AsyncStorage 做最薄的一层封装

### 领域层

`src/domain/models` 定义训练模式和进度相关的数据模型。

`src/domain/services/progressService.ts` 负责：

- 生成默认进度
- 标准化本地存储数据
- 限制周目标范围
- 切换某模式当日完成状态
- 清理过期历史数据

`src/domain/services/dashboardService.ts` 负责：

- 计算今日完成进度
- 计算本周训练进度
- 计算累计场次
- 计算当前连练和最佳连练
- 生成最近 7 天节奏图
- 生成今日推荐训练列表

### 功能层

`src/features/dashboard` 负责首页展示，包括：

- Hero 卡片
- 今日安排卡片
- 本周节奏卡片
- 训练模式卡片列表

`src/features/mode-detail` 负责训练模式详情页，展示：

- 模式说明
- 今日训练步骤
- 检查清单
- 完成标准
- 打卡入口

### 内容层

`src/data/seed/trainingModes.ts` 是当前最核心的产品内容入口。它定义了全部 8 个训练模式的元数据，也是之后接入真实题流时最可能继续保留的模式描述层。

## 这个项目现在到底是什么

如果用更直接的话说：

- 它不是一个完整的题库 App
- 它不是一个带账号系统的学习平台
- 它不是一个已经做完的 JLPT 产品

它现在是：

- 一个已经完成视觉和结构骨架的 N2 训练 App 原型
- 一个以“今日节奏”和“训练模式”组织学习体验的产品雏形
- 一个适合继续往每个模式里接真实学习内容的前端底座

## 产品价值判断

这个方向本身是成立的，因为 JLPT 备考里最常见的问题不是“完全没有资料”，而是：

- 不知道今天该练什么
- 学习节奏不稳定
- 容易只做擅长的项目
- 错题没有被持续回收

这个项目当前已经在尝试解决前两个问题：

- 用“今日安排”降低选择成本
- 用“本周节奏”和“连练”强化持续性

下一阶段如果把“错题回收”和“真实训练内容”接上，就会开始真正解决后两个问题。

## 建议的开发路线图

下面这条路线是基于当前代码结构和产品状态做出的建议，重点是尽快把原型推进成“能真正使用”的第一版。

### 第一阶段：把说明页变成真正的训练页

优先给至少 2 个模式接入真实训练流程，建议从：

- `grammar_drill`
- `vocab_drill`

开始。

原因：

- 这两个模式最容易定义统一题型
- 最容易验证“做题 -> 解释 -> 完成 -> 记录”的基本闭环
- 对用户感知价值最直接

这一步建议先定义统一内容 schema，例如：

- `id`
- `modeId`
- `prompt`
- `choices`
- `answer`
- `explanation`
- `tags`
- `source`

训练模式继续保留为“模式元数据层”，题目内容单独放在新的数据文件或内容仓储中。

### 第二阶段：从手动打卡升级为自动完成记录

现在的完成方式是用户手动点击“已完成”。这对原型足够，但对真实产品不够。

更合理的规则应该是：

- 用户完成一轮训练后自动记一次 session
- 用户退出中途训练时不记完成
- 学习包模式和做题模式可以有不同完成判定

这一步完成后，进度系统会从“勾选型习惯追踪”变成“真实训练记录”。

### 第三阶段：接入错题闭环

项目现在已经有“文法错题回收”和“词汇错题回收”两个模式，但它们还只是静态占位。

建议尽快建立错题数据结构，至少记录：

- 题目 id
- 错误类型
- 首次错误日期
- 最近复习日期
- 错误次数
- 是否已掌握

一旦这部分做起来，错题回收模式就会从文案页面变成真正有价值的复习入口。

### 第四阶段：升级今日推荐逻辑

当前“今日安排”逻辑比较简单，本质上只是从未完成模式中取前 3 个。

后续建议引入规则驱动的推荐，例如：

- 最近两天没练过的模式优先
- 错题积压较多的模式优先
- 临近考试时提高读解和听力权重
- 最近过度训练的模式适当降权

这一步会显著提高产品的“像教练”的感觉。

### 第五阶段：确认内容闭环后再考虑账号和同步

当前没有后端不是问题，反而是合理取舍。

在真实题流、错题回收、推荐逻辑还没稳定之前，不建议过早上：

- 登录
- 云同步
- 远程题库管理
- 多端账号系统

更合理的顺序是：先把单机体验跑通，再决定是否需要服务端。

### 第六阶段：补工程化能力

在产品闭环基本稳定后，再补以下工程内容最划算：

- 为 `progressService` 和 `dashboardService` 补单元测试
- 为训练内容 schema 补校验
- 为本地存储结构补版本迁移机制
- 补完整 README 和开发说明
- 补内容导入与维护流程

## 建议的版本拆分

如果要把这个项目推进成一个真正可用的第一版，我建议用下面的节奏：

### v0.1

- 保留现有 dashboard
- 接入 `grammar_drill` 和 `vocab_drill` 的真实训练流
- 训练完成后自动记录 session

### v0.2

- 建立错题记录结构
- 打通 `review_wrong` 与 `vocab_review_wrong`
- 完成基础复盘闭环

### v0.3

- 升级今日推荐逻辑
- 让首页安排真正个性化

### v0.4

- 评估账号系统与云同步
- 扩展题库和内容维护能力
- 强化工程化和测试

## 当前最值得优先做的事

如果只选两件最重要的事情，我会建议优先做：

1. 真实题流
2. 错题回收闭环

原因很简单：

- 没有真实题流，产品只能看，不能真正练
- 没有错题回收，产品就缺少长期复习价值

只要把这两部分接起来，这个项目就会从“漂亮的训练面板”升级成“真正能帮助备考的工具”。

## 总结

这个仓库当前最准确的定位是：

**一个已经完成训练框架、进度系统和主界面原型的 JLPT N2 备考 App 前端底座。**

它现在最强的部分是：

- 训练模式设计
- 日常节奏表达
- 本地打卡反馈
- 清晰的前端分层

它现在最欠缺的部分是：

- 真实训练内容
- 错题机制
- 自动完成闭环
- 个性化推荐

下一步不需要先做更复杂的 UI，也不需要先做后端，最务实的方向是把模式页真正做成可训练、可记录、可复盘的页面。
